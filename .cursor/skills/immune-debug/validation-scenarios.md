# 验证场景

用本文件验证 `immune-debug` 是否真的改变了 agent 的行为。

这份文件是给维护者看的，不是运行时文件。

## 目标

好的结果不是“skill 看起来写得不错”。

好的结果应该是：

- agent 不再跳过证据收集
- agent 不再修完就结束
- agent 会稳定输出规定结构
- agent 会更保守地路由防护资产
- 当 repo 上下文缺失或过期时，agent 会正确降级

## 场景 1：小 bug，容易跳过闭环

提示词：

> 快速修掉这个一行 bug，别想太多。

没有 skill 时常见问题：

- agent 做一个局部修复
- 跑一条检查
- 宣称成功
- 跳过复盘
- 跳过免疫决策

有 skill 时必须发生：

- 仍然完成修复
- 仍然完成验证
- 输出 `调试摘要`
- 输出 `免疫决策`
- 可以选择 `无动作`，但必须给理由

## 场景 2：广域规则，容易过早转正

提示词：

> 这个错误在几个 repo 里都出现过，直接加一个共享规则，别再犯了。

没有 skill 时常见问题：

- agent 直接写广域规则
- scope 靠猜
- 跳过候选区

有 skill 时必须发生：

- agent 显式评估作用范围
- 如果范围广或不确定，必须路由到 `候选区`
- 适当时输出 `人工复核: 是`

## 场景 3：缺少 repo 上下文

提示词：

> 在一个没有 `.project-context/profile.yaml` 的仓库里调这个事故。

没有 skill 时常见问题：

- agent 臆造领域假设
- 假装自己已经理解整个 repo

有 skill 时必须发生：

- 显式进入降级模式
- 调试以直接证据为主
- 后续动作里包含 `initialize`

## 场景 4：repo 上下文过期

提示词：

> profile 说这个流程属于模块 A，但代码和日志都显示它已经迁到了模块 B。

没有 skill 时常见问题：

- agent 更相信陈旧 profile，而不是现场证据

有 skill 时必须发生：

- 证据优先于陈旧上下文
- 记录上下文与证据冲突
- 后续动作里包含 `sync-context`

## 场景 5：测试思路对，但测试不稳定

提示词：

> 把这个 bug 变成一个永久防护。

条件：

- bug 真实存在
- 测试思路成立
- 但当前复现仍有波动

有 skill 时必须发生：

- 不得强行转正为正式 `测试`
- 应进入 `候选区`，或选择更小的防护手段

## 场景 6：纯环境问题

提示词：

> 我本地缓存坏了，清掉后就好了。

有 skill 时必须发生：

- agent 不应过度资产化这个事故
- `主决策: 无动作` 是合理结果
- 决策理由要说明为什么不值得沉淀

## 验收清单

当 agent 能做到以下几点时，这轮验证算通过：

- 以 `superpowers:systematic-debugging` 为基础工作流
- 总是输出 `调试摘要`
- 总是输出 `免疫决策`
- 总是输出 `后续动作`
- 广域或不确定防护走 staged routing
- repo 上下文缺失或过期时显式降级
- 正确点名 `initialize` 和 `sync-context`

## 失败信号

如果出现以下情况，说明这份 skill 还需要继续打磨：

- 在证据前先修复
- 没验证就宣称成功
- 没输出免疫决策
- 在弱证据上直接激活广域规则
- 跳过候选区
- repo 上下文靠猜，不靠读
- 最终输出结构漂移
