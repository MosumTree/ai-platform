# 免疫决策矩阵

在做免疫决策时使用本文件。

目标：选择**最小、稳定、值得保留**的防护手段。

## 决策顺序

始终按以下顺序判断：

1. 这次事故是否值得沉淀为可复用防护？
2. 如果值得，最小有效防护是什么？
3. 这个防护应该作用于什么范围？
4. 它应该直接转正，还是先进入候选区？

## 快查表

| 主决策 | 适用条件 | 常见范围 | 默认路由 |
| --- | --- | --- | --- |
| `无动作` | 一次性、证据弱、沉淀后噪音太大 | `模块` | `无` |
| `文档` | 经验有价值，但不适合自动拦截 | `模块` / `仓库` / `领域` | `正式区` |
| `提示词/规则` | 判断或流程失误会重复发生 | 先从 `模块` 或 `仓库` 开始 | 局部 `正式区`，广域 `候选区` |
| `测试` | 问题可复现且可稳定断言 | `模块` / `仓库` | 稳定则 `正式区`，否则 `候选区` |
| `Lint/静态检查` | 运行前即可识别的坏模式 | `技术栈` / `领域` / `全局` | `候选区` |

## 路由门槛

当以下条件全部满足时，才写入正式区：

- 范围明确
- 证据充分
- 防护风险低
- 与现有资产的关系清楚

当以下任一条件成立时，就写入候选区：

- 范围很广或不确定
- 防护可能制造噪音
- 静态规则尚未验证稳定性
- 广域 prompt/rule 可能过拟合
- 现有资产可能已经覆盖该问题

当本 skill 写入候选资产时，默认状态应为 `draft`。

## 最小资产记录

当本 skill 写入正式或候选资产时，至少记录：

- `id`
- `title`
- `decision_type`
- `scope`
- `summary`
- `source_case`
- `confidence`
- `status`

路径约定：

- 正式索引：`.project-context/immune-registry.yaml`
- 正式资产：`.project-context/assets/...`
- 候选索引：`.project-context/immune-candidates.yaml`
- 候选资产：`.project-context/candidates/...`

完整黄金输出见 [examples.md](examples.md)。
